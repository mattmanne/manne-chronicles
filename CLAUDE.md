# The Manne Chronicles — Architecture Notes

This is the developer-facing companion to `README.md` (which is written for players). Read this before making structural changes.

The Manne Chronicles is the umbrella app name; "Resonance" and "Manlandia" below are the names of two built-in playable worlds inside it (plus unlimited user-created custom worlds) — not alternate names for the app itself.

## Tech stack

- **Vercel** — hosting + serverless functions (`api/*.js`), auto-deploys on push to `main`
- **Groq API** — Llama 3.3 70B, called from `lib/gemini.js` (file is named `gemini.js` for historical reasons — it does **not** call Google's Gemini API; don't rename it, several places assume this path)
- **Upstash Redis** — REST API only, no SDK (`lib/redis.js`); this is the only persistence layer
- **One npm dependency**: `web-push`, added deliberately for push notifications — everything else still uses native `fetch` (Redis, Groq). Web Push requires implementing a real cryptographic protocol (VAPID JWT signing + ECDH/HKDF/AES-128-GCM payload encryption, RFC 8291/8292) to actually send a notification, unlike Redis/Groq which are just plain REST calls — hand-rolling that protocol is easy to get subtly wrong with silent failures (notification just never arrives) and no way to verify correctness without a real device round-trip. That's different enough from "avoid an SDK when fetch works fine" to justify the one exception. Test tooling (`node:test`) is still Node-built-in.
- **Frontend**: plain HTML/CSS/JS, no bundler, no framework. `public/index.html` loads `pure.js` then `game.js` as ordinary global-scope `<script>` tags (in that order — `game.js` calls functions defined in `pure.js`).

## World routing (`lib/worldconfig.js`)

Every API handler starts by calling `getWorldConfig(req.query.world)`, which returns `{ id, type?, key, getInitialState, buildSystemPrompt }`:

| `world` param | Resolves to | Redis key | Gamestate / prompt |
|---|---|---|---|
| `"resonance"` | built-in | `resonance:gamestate` | `lib/gamestate.js` / `lib/prompt.js` |
| `"manlandia"` | built-in | `manlandia:gamestate` | `lib/gamestate-manlandia.js` / `lib/prompt-manlandia.js` |
| `c_<anything>` | custom campaign | `campaign:<id>:gamestate` | `lib/gamestate-custom.js` / `lib/prompt-custom.js` |
| anything else / missing | falls back to `"resonance"` | — | — |

Custom worlds carry their own config (`name`, `theme`, `playerCount`, `adult`) inside `gameState.worldConfig`, not in the static `getWorldConfig()` return value — that config only exists once a campaign has actually been created and stored.

`lib/worldconfig.js` also exports `buildWorldStatePayload(worldConfig, gameState)` — the single source of truth for "which `worldState` fields does this world type expose to the client," shared by `api/gm.js`'s response and `api/poll.js`'s poll payload (previously hand-duplicated in both files; every new field like `npcs`/`inventory`/`last_actor` had to be added to both separately before this existed).

**Custom campaign IDs** (`api/campaigns.js`'s `create` action) are `c_<timestamp>_<8-char random hex>`, e.g. `c_1784469655708_a3f9c1d2`. Before 2026-07-19 this was just `c_${Date.now()}` — a bare millisecond timestamp with no collision protection at all. Two creates landing in the same millisecond (a double-tap on "Create World," a network retry, two devices creating at nearly the same instant) would get the *identical* id and, since the Redis key is derived purely from it (`campaign:<id>:gamestate`), would silently share one gamestate from then on — both campaigns' session logs, NPCs, and narration mixing together with no error or warning to anyone. Found while investigating a "worlds bleeding into each other" report (that specific incident turned out to be something else — see the Resolved section of `project_resonance_backlog_decisions.md` — but the risk itself was real and worth closing regardless). The random suffix guarantees uniqueness independent of timing; the timestamp prefix is kept only for rough chronological readability when eyeballing raw Redis keys — `createdAt` in `campaigns:index` is the actual source of truth for ordering.

## Redis key structure

- `resonance:gamestate`, `manlandia:gamestate` — the two built-in worlds' full state
- `campaign:<id>:gamestate` — one per custom campaign
- `campaigns:index` — array of `{ id, name, subtitle, playerCount, adult, createdAt, status }`, the list rendered in the world selector. `status` is `"active"` or `"archived"` — set via `action: "archive"`/`"unarchive"` in `api/campaigns.js`, reversible, and purely a world-selector visibility toggle (`public/game.js`'s `renderCampaignList()` hides archived entries by default behind a "Show N archived worlds" toggle). Campaigns created before this field existed have no `status` at all — treated as active everywhere (`c.status !== "archived"`), not backfilled.

**Known quirk**: deleting a campaign (`api/campaigns.js`, `action: "delete"`) only removes its entry from `campaigns:index` — the `campaign:<id>:gamestate` key itself is never deleted, so it becomes an orphaned, unreachable key in Redis. Harmless (small, never read again) but worth knowing if you're ever auditing Redis usage. See `[[project_data_safety]]` in memory re: Redis data being precious — don't add cleanup logic here without checking that memory note first. Archiving (above) doesn't have this problem at all, since nothing is ever deleted.

**Delete requires typing "delete" to confirm** (2026-07-05, after a real misclick archived two campaigns) — `#delete-confirm-overlay`'s "Delete Forever" button (`public/game.js`'s `setupDeleteConfirm()`) stays `disabled` until the player types the word "delete" (case-insensitive, trimmed) into `#delete-confirm-input`; the click handler also re-checks the typed value defensively rather than trusting the `disabled` attribute alone, same "don't trust client-side-only gating" instinct used elsewhere. The overlay is reused across different campaigns, so `deleteCampaign()` resets the input and re-disables the button every time it reopens — a stale "delete" typed for a previous campaign must never carry over and silently arm the button for a different one. Archive (the reversible alternative, already one click via the 🗄/↩ toggle) is deliberately left frictionless — only the permanent, unrecoverable action got harder to trigger by accident.

## GM bracket-tag notation

The GM's raw LLM response is plain narration plus optional bracket tags on their own lines at the end. `api/gm.js` orchestrates the request/response cycle after the response comes back from `lib/gemini.js`; the actual tolerant regex logic lives in `lib/gm-tags.js` (pure functions, unit-tested directly — see `tests/gm-tags.test.js`), and the tag-to-state-mutation wiring (`applyStateTags()` and the three location matchers) lives in `lib/apply-state-tags.js` — extracted out of `api/gm.js` once it became the single highest-churn function in the app (every new tag lands there). Every prompt file (`lib/prompt.js`, `lib/prompt-manlandia.js`, `lib/prompt-custom.js`) instructs the model to emit whichever of these apply, in a "STATE NOTATION" block.

| Tag | Worlds | Effect |
|---|---|---|
| `ROLL:STAT[:ADVANTAGE]` | all | Not a bracket tag — a bare line, in theory. Triggers the dice-roll UI; client resubmits the result as `type: "roll_result"`. `lib/gm-tags.js`'s `extractRoll()` tolerates an optional `[...]` around `STAT`, a space after the colon (live: `"ROLL: [AGILITY]"`), and any case. This bit the app for real, three times now: first with bracketed stats silently failing to trigger a roll, then the colon-space variant slipping past that fix, then — most fundamentally — the model embedding the whole trigger **mid-sentence instead of alone on its own line** at all (live, Dark Wars: `"Note: ... The roll is: ROLL: WISDOM"`), which a line-anchored regex could never match no matter how tolerant its edges were. `extractRoll()` no longer requires the tag to be alone on a line at all — it scans every `ROLL:` occurrence anywhere in the text and uses the first one that resolves to a real stat (skipping past the model's own bogus explanation-prefixed `"ROLL: This looks tricky!..."` along the way), then strips *every line* that mentions `ROLL:` anywhere in it (not just lines starting with it) from the display. Don't tighten this back up without re-checking `tests/gm-tags.test.js`'s live-captured fixtures. `STAT` also accepts genre-standard D&D/DW synonyms — `STAT_SYNONYMS` in `lib/gm-tags.js` maps things like `PERCEPTION`/`WISDOM`/`INTELLIGENCE`→Acuity, `STRENGTH`/`MIGHT`→Force, `DEXTERITY`/`SPEED`/`STEALTH`→Agility, `CONSTITUTION`/`RESOLVE`/`COURAGE`→Will, `CHARISMA`/`PERSUASION`→Presence — because the model keeps reaching for `PERCEPTION` specifically (confirmed live, twice, in the "Dark Wars" custom campaign) despite every prompt naming it by name as a forbidden example. A word not in the map still safely falls through and keeps scanning for another occurrence, same as before. **Also accepts an optional character name as a second colon-separated token in either order** — `ROLL:FEN:AGILITY` or `ROLL:AGILITY:FEN` — resolved by `extractRoll()`'s second return value, `rollPlayer`. This exists because a merged multi-character turn (see "Wait-for-all-players turn gating" below) has no single implicit submitter to attribute the roll to anymore; every prompt file now tells the model to name the roller whenever more than one character acted in the same turn. With no name present, `rollPlayer` is `null` and `api/gm.js` falls back to the single submitter — the original, unchanged behavior. **Also tolerates the trigger being entirely absent** — a 2026-07-05 live playtest (all 6 personas, across 2 separate campaigns) confirmed the model sometimes asks for a roll in plain prose with no `ROLL:` anchor anywhere at all (`"Roll a FORCE to see how well Kestra can hold her ground"`, `"Rolling Bram's Force for the attack"`) despite every prompt file explicitly instructing it to always include the literal trigger line. Since no amount of tolerance on the `ROLL:` token itself could ever catch a response that doesn't contain that token, `extractRoll()` now falls back to a natural-language scan (only when the primary `ROLL:` scan finds nothing) for `Roll(ing) [the/a/an/your/Name's] STATWORD` phrasing, resolving the stat via the same `STAT_SYNONYMS` map and the roller via the same name-lookup as the explicit-tag path. Guarded against false-firing on resolution narration ("With a roll of 13 on Acuity...") by requiring a real stat word immediately after "Roll"/"Rolling" and excluding "roll of" — verified against every real "roll"-mentioning GM entry in production with zero false positives. Unlike the bracket-tag paths, this natural-language match is **not** stripped from `clean` — it's real narration a player should read, not mechanical tag syntax. |
| `[LOCATION: Name]` | all | Updates `worldState.location`; if the name matches a known location, adds it to `visited_locations` (deduped). Resonance/Manlandia match against a fixed list of known place names (`matchResonanceLocationId()`/`matchManlandiaLocationId()` in `lib/apply-state-tags.js`) for their curated, pre-drawn maps. Custom worlds have no such list — the GM invents arbitrary locations per campaign — so `matchCustomLocationId()` just uses the location's own text as its id (exact-string dedup, not fuzzy). This is what powers `public/game.js`'s "quest trail" graph for custom worlds (see "Quest trail (custom-world map)" below) — before this, custom worlds' `visited_locations`/`location_scars` existed in the data model but were always empty, since the old matcher unconditionally returned `null`. **Prompt reinforcement added 2026-07-06**: a live playtest found the model narrating a party settling into a genuinely new location (a town, a tavern) for several turns with the tag never firing at all — all three prompt files' LOCATION CHANGE instruction now includes a concrete ✗ WRONG / ✓ RIGHT example targeting that exact failure, same treatment as the `ROLL:`/COMBAT drift fixes. Prompt-adherence only, no parser-side fallback attempted (same reasoning as combat — no narrow, reliable grammatical signal for "a real move happened"). |
| `[SCAR: Location: Label]` | all | Adds `{ id, label }` to `worldState.location_scars` (deduped by id+label) |
| `[SUGGESTIONS: a \| b \| c]` | all | Parsed by `lib/suggestions.js`, returned as the `suggestions` array; forced to `[]` server-side whenever `needsRoll` is true or `type === "roll_result"`. **Also accepts the tag wrapped in parentheses instead of brackets** — live (2026-07-06, this app's own deployment-verification calls, twice): `"(SUGGESTIONS: a \| b \| c)"` — the original bracket-only regex silently dropped it, leaving `suggestions: []` despite the model clearly intending a discrete-choice list. `public/pure.js`'s `stripGMTags()` has the matching tolerance so it doesn't leak into display either. |
| `[XP N: +amount]` | manlandia, custom | Awards bonus growth XP (N = player number) via `extractXpBonuses()`, applied through `applyXpGain()` — see "Character growth (XP)" below for the full system. Not taught to Resonance's prompt at all (no growth system there). |
| `[CONCLAVE AWARENESS: X → Y]` | resonance | `worldState.conclave_awareness = Y`. `extractCounterUpdate()` tolerates an ASCII `->` in place of `→`, extra whitespace, and case — never empirically observed failing, but cheap insurance |
| `[DISSONANCE: X → Y]` | resonance | `worldState.fen_dissonance_awakening = Y`. Same tolerance as above. |
| `[LYRA\|FEN: OldHarm → NewHarm]` | resonance | Sets that character's `harm`, via `extractResonanceHarmUpdates()` — same arrow tolerance, plus the harm word is normalized case-insensitively against `HARM_LEVELS` (`lib/gamestate.js`) and dropped entirely if it isn't a real harm level, so a typo can't silently corrupt `recover_harm`'s `HARM_LEVELS.indexOf()` lookup |
| `[ABILITY FEN\|LYRA: ability_name]` | resonance | Sets that boolean field true on the character (or decrements `lyra.magic_uses_remaining` when `ability_name === "magic"`). Not hardened against typos in `ability_name` — no live evidence of that failing yet, unlike everything else on this list |
| `[VILLAIN AWARENESS: X → Y]` | manlandia, custom | `worldState.villain_awareness = Y`. Same `extractCounterUpdate()` tolerance as CONCLAVE AWARENESS. |
| `[CURSE: X → Y]` | manlandia, custom | `worldState.curse_level = Y`. Same tolerance. |
| `[CHARACTER N: OldHarm → NewHarm]` | manlandia, custom | Sets `characters.playerN.harm`, via `extractCharacterHarmUpdates()`. **Also accepts the hero's actual name in place of "CHARACTER N"** — confirmed live in a real campaign (`"[Globak: Unhurt → Scratched]"` for a hero who should have been `CHARACTER 2`; the model prefers narrating a named hero over an anonymous slot number despite the prompt's explicit instruction not to). Same harm-word normalization as the Resonance harm tag. Also tolerates trailing commentary after the new harm word before the closing bracket — live: `"[CHARACTER 1: Scratched → Scratched, no change]"`, which the stricter original regex silently rejected (harmless that time since old/new were identical, but would've dropped a real change the same way). `public/pure.js`'s `stripGMTags()` has a matching generic `[Name: Harm → Harm]` strip so the raw tag doesn't leak into what the player sees either way. **Also tolerates a `(Name)` parenthetical inserted between the number and the colon, plus a literal `Harm:` label before the harm word** — live (2026-07-05 playtest): `"[CHARACTER 1 (Kestra): Harm: Unhurt]"`. Neither padding matched the original regex at all (the colon has to directly follow the digit), so a real harm change written in this shape would have silently no-op'd; `stripGMTags()` was updated with the matching tolerance so it doesn't leak either. |
| `[ABILITY N: used]` | manlandia, custom | Sets `characters.playerN.ability_used = true`, via `extractAbilityUsedKeys()`. Tolerates the model padding in the ability's name (live: `"[ABILITY 1: Lucky Break used]"`) as long as the word "used" appears — but a negative lookahead means `"[ABILITY 1: Lucky Break - not used]"` (also seen live) correctly does **not** fire. Once a hero has unlocked bonus abilities (see "Character growth" below), that same padded name is also how a *bonus* ability's use is told apart from the starting one: if it names one of the hero's `bonus_abilities` (matched loosely against the id's own words), the use is recorded in `bonus_abilities_used` instead of the shared `ability_used` boolean — no name, or a name that doesn't match a bonus ability, keeps the original starting-ability behavior. **Also tolerates the same `(Name)` parenthetical after the number** as the CHARACTER N tag above — live: `"[ABILITY 1 (Kestra): Protect Friend used]"`, confirmed live to silently fail (`ability_used` stayed `false`) before this tolerance was added; `stripGMTags()` was updated to match so it doesn't leak either. |
| `[STONE FOUND: stone_id]` | manlandia only | Adds to `worldState.stones_found` (deduped); `stone_id` must be one of `STONE_IDS` in `lib/gamestate-manlandia.js` |
| `[OBJECTIVE: description]` / `[OBJECTIVE COMPLETE: description]` | all | Generalizes the stone tracker's fixed-ID checklist to arbitrary free-text quest goals, via `extractObjectiveUpdates()`. Adds `{ text, done: false }` to `worldState.objectives`, deduped against existing entries; `OBJECTIVE COMPLETE` marks the closest match done. Matching is **fuzzy** (`objectivesMatch()` — significant word overlap, not exact text), since the model isn't expected to phrase a completion identically to how it phrased the original goal — deliberately permissive, and likely to need a follow-up pass once real transcripts show how the model actually phrases these, same as every other tag on this list. |
| `[CLUE: description]` / `[CLUE RESOLVED: description]` | all | "Leads" — investigation-shaped state (open questions, contradictions, unanswered suspicions), separate from Objectives' goal-shaped tracking ("find the sword" vs. "who lied about the sword"). Deliberately reuses `objectivesMatch()` verbatim for dedup/fuzzy-resolve rather than inventing a second matcher — same paraphrase-tolerance need. Adds `{ text, done: false }` to `worldState.clues`, via `extractClueUpdates()`. Rendered as a "🔍 LEADS" panel on the Map tab (`renderClues()`), reusing the same `.objective-item`/`.objective-check` CSS as Objectives. No "this clue contradicts clue X" relationship yet — deferred until real transcripts show it's worth the added risk, same posture as everything else in this table. **Provisional**: naming ("Leads") and scope (universal, not Resonance-only) are recommendations pending Matt's confirmation — see `project_resonance_backlog_decisions.md`. |
| `[NPC: Name: Description]` | all | Adds `{ name, description }` to `worldState.npcs`, via `extractLorebookUpdates()`. Shared across the whole party, same as objectives. Dedup is by name only (case-insensitive exact match, not fuzzy word-overlap like objectives) — NPC names are proper nouns, so exact-ish is the right bar. No "update an existing NPC's description" tag yet — a repeat name is just skipped; first-pass scope, same as every other tag here until real transcripts show whether that's ever actually needed. Rendered as a "📖 WHO'S WHO" panel on the Map tab (`renderLorebook()` in `public/game.js`). |
| `[ITEM FOUND: description]` | manlandia, non-adult custom | Adds to the shared `worldState.inventory` array (deduped, case-insensitive), via `extractSharedItemAdditions()`. Rendered as a "🎒 PARTY INVENTORY" panel on the Map tab (`renderInventory()`). **Prompt reinforcement added 2026-07-06** (same treatment/reasoning as `LOCATION`/`COMBAT` above): a live playtest found clear, explicit pickups ("I scoop up the key...") narrated with no matching tag, `worldState.inventory` staying `[]` all session. |
| `[ITEM N: description]` | adult-flagged custom only | Adds to `characters.playerN.inventory` (deduped per-character), via `extractCharacterItemAdditions()`. **Also accepts the hero's actual name in place of "ITEM N"**, same tolerance as `CHARACTER N` harm. Rendered inline on that hero's own character card (`renderCharInventory()`), not the shared Map-tab panel. Same 2026-07-06 prompt reinforcement as `ITEM FOUND` above. |
| `[ITEM FEN\|LYRA: description]` | resonance | Adds to that character's own `inventory` array (deduped), via `extractResonanceItemAdditions()`. Resonance is always adult, so it only ever uses the per-character form — there's no `worldState.inventory` field for Resonance at all. Same 2026-07-06 prompt reinforcement as `ITEM FOUND` above. |
| `[COMBAT START: Enemy A, Enemy B]` | all | Sets `worldState.combat.active = true`, adds `{ name, harm: "Unhurt", defeated: false }` for each new enemy name via `extractCombatStart()`. Not line-anchored (same lesson as `extractRoll`'s mid-sentence fix); tolerates "and" in place of a comma and strips leading list-numbering per name. See "Combat" below for the full design. |
| `[ENEMY: Name: OldHarm → NewHarm]` | all | Updates that enemy's harm via `extractEnemyUpdates()` — fuzzy name match against the tracked, not-yet-defeated enemy list (same "model shortens/renames" tolerance as `CHARACTER N` harm), plus an arrow-less `[ENEMY: Name: Harm]` fallback. If the "new value" is a defeat synonym (`defeated`, `destroyed`, `fled`, etc.) instead of a real harm word, it's treated as a defeat rather than a silently-dropped unrecognized harm value — the single most likely way the model ends a fight in prose. |
| `[ENEMY DEFEATED: Name]` | all | Marks that enemy `defeated: true`, via `extractEnemyDefeated()`. Once every tracked enemy is defeated, `applyStateTags()` auto-sets `combat.active = false` without waiting for `COMBAT END` — see below. |
| `[COMBAT END]` | all | Explicitly ends the fight via `extractCombatEnd()`. Enemies are left in the array (not wiped), just hidden once `active` is false, so the tail of the fight's narration can still reference them. |

**Inventory is split by adult/kid, not just by world**, matching `lib/growth.js`'s existing XP-pacing split: kid-friendly games (Manlandia, non-adult custom) track one **shared party inventory** on `worldState` — fits how loot usually works in those games, the party finds things together. Adult games (Resonance, adult-flagged custom) track **per-character carried items** instead — fits the "everyone's aware of each other" table dynamic that also justified making Bonds adult-only. For custom campaigns this is decided once, at creation time, from `worldConfig.adult` (`lib/gamestate-custom.js`'s `getInitialStateCustom()`) — a kid custom campaign's characters never get an `inventory` field at all, and an adult one's `worldState` never gets one either, so neither shape carries a permanently-dead field the way `known_allies`/`known_enemies`/`dead_npcs` do elsewhere in this file. `lib/apply-state-tags.js`'s `applyStateTags()` still checks both the `ITEM FOUND` and `ITEM N` patterns unconditionally in the manlandia/custom branch rather than gating on the adult flag — each prompt file only ever teaches the model the one tag matching its own world, so in practice only one ever fires; this is the same "parse whatever shape shows up" tolerance as every other tag, not a real ambiguity. One side effect: the lazy `if (!gameState.worldState.inventory) ... = []` init in that shared code path means an adult custom campaign's `worldState.inventory` does exist at runtime after its first turn — just permanently empty, since nothing ever pushes into it. Harmless, consistent with this app's existing tolerance for that class of unused-but-harmless field.

Never trust these tags to be well-formed — the model can omit them, duplicate them, or get creative with spacing. Every parser here is defensive (regex non-matches are silently ignored, dedup checks before pushing). When investigating a "the GM said X happened but the state didn't update" report, the fastest diagnostic is pulling the real campaign's stored `sessionLog` via `GET /api/state` and grepping for `[` — that's how every bug in this table was actually found (real transcripts, not speculation).

## Combat

This app's combat has always been Dungeon-World-style — one roll resolves a whole exchange, the 10+/7-9/4-6/2-3 result bands already encode both success and retaliation — and that doesn't change here. What `worldState.combat` adds is just **enemy state that persists across several exchanges within one fight**, instead of every fight (including bosses) resolving in a single roll. There is deliberately no initiative queue, no positioning system, and no per-round rolls — that would fight the app's whole existing philosophy, not extend it.

- **Universal shape, not kid/adult split** — unlike inventory/Bonds/XP, `worldState.combat = { active, round, enemies: [{ name, harm, defeated }] }` is the same shape for every world type. A goblin's harm track doesn't need to differ by audience the way carried items or relationship mechanics do; only the *narration tone* around combat differs, which lives in the prompt files, not the data model. Enemy harm reuses the existing `HARM_LEVELS` scale and `normalizeHarm()` — no new enemy-specific scale.
- **`round` is server-derived, never model-supplied** — `api/gm.js` increments it by 1 whenever a `roll_result` is processed while `combat.active` is true, and that's the only place it changes. The model is never asked to count rounds itself, which sidesteps an entire category of drift risk for this one value.
- **Kid-world harm cap**: Manlandia and non-adult custom campaigns cap a hero's harm at `Broken` during an active fight — `[CHARACTER N: ... → Dying]` gets silently rewritten to `Broken` in `lib/apply-state-tags.js`, using the same `isAdultWorld()` check as everything else that needs it. A sustained multi-round fight is far more likely to actually walk a hero's harm all the way down the track than the old one-roll model ever was, and letting an 8-year-old's hero reach Dying mid-fight was judged a real experience question worth a deliberate answer, not a default. Resonance and adult-flagged custom worlds are untouched — no cap. The kid-facing prompt files also tell the model not to narrate past Broken during a fight, so the narration and the (occasionally silently-corrected) state stay in sync in the common case.
- **Ending a fight has three independent paths**, since `[COMBAT END]` alone (a pure "stop" signal with no natural sentence to hang it on) is expected to be the single tag in this whole table most likely to just never get emitted: (1) the explicit `[COMBAT END]` tag, (2) auto-detection once every tracked enemy has `defeated: true`, checked on every turn in `applyStateTags()`, and (3) a manual "🏳 End Combat" button in the client's combat tracker, wired to a new `action: "end_combat"` on `POST /api/state` — no LLM round-trip, same pattern as `recover_harm`. This covers the residual gap where a fight ends by retreat or negotiation rather than defeating everyone.
- **UI**: `#combat-tracker` in the Story tab (not a new tab, not the Map tab — combat is a transient, in-the-moment concern, not a persistent reference collection), rendered by `renderCombatTracker()` in `public/game.js`, wired into the same `updateCharacterUI()` call every other live-updating panel already goes through. Hidden entirely when `combat.active` is false.
- **No changes to the roll pipeline** — `sendToGM()`'s roll → animate → auto-resubmit chain, `animateRoll()`'s dice math, and the roll-deferral behavior (combat tags are deferred while a roll is pending, exactly like every other tag, with zero special-casing needed since they go through the same `applyStateTags()` function) all already worked per-exchange before this existed.
- **Combat status is echoed back to the model every turn** — `lib/prompt-shared.js`'s `buildCombatStatusBlock(ws)`, wired into all three prompt files' `CURRENT CAMPAIGN STATE` block, added 2026-07-06 after a live playtest found two related failures: combat sometimes never engaging at all despite an unambiguous physical fight, and — even once it had — enemy harm/defeat drifting out of sync with the narration (a narrated killing blow leaving the tracked enemy still `Unhurt, not defeated`). Character harm was always shown back to the model every turn via each prompt's `charLine()`; enemy harm never was, so the model had no persistent anchor for what it still owed the tracker. This block shows "No active fight right now" (plus a reminder of when to start one) when `combat.active` is false, or the live round number and every tracked enemy's current harm/defeat state when it's true — same idea as `charLine()`, just for enemies. All three prompt files' COMBAT instructions also gained a concrete ✗ WRONG / ✓ RIGHT example pair targeting the exact failure phrasing observed live (a clear physical clash narrated with zero tags). Both are prompt-adherence mitigations, not a parser-level fix — unlike the `ROLL:` drift fix, there's no narrow, reliable grammatical signal for "a fight is starting" the way there is for "a roll is being requested," so a regex-based fallback was deliberately not attempted here; if this doesn't sufficiently close the gap, revisit via `npm run check-drift` once more real combat transcripts exist.

## Quest trail (custom-world map)

Custom worlds have no fixed geography to pin a real map to — the GM invents arbitrary locations per campaign, with no coordinates and no artist available to hand-draw one per world (unlike Resonance/Manlandia's curated, pre-drawn maps). `renderQuestGraph()` in `public/game.js` (replacing the older plain-text "journey trail" list) lays `visited_locations` out as a deterministic snake-path graph — row/column purely computed from each location's index in the array, alternating left-to-right then right-to-left per row, board-game-trail style. This keeps a location's position stable as the campaign grows, since `visited_locations` only ever appends (deduped by exact text in `matchCustomLocationId()`, never reordered). Reuses the same pulse-ring/scar-mark SVG primitives as Manlandia's real map for visual consistency, just with a single node color (no location "type" categories exist for custom worlds) and truncated labels with a native `<title>` tooltip for the full name, since custom location names are free text and can be much longer than Manlandia's curated short names. **This is a provisional design call, not fully locked in** — it ships the "a non-geographic graph is an acceptable stand-in for a literal hand-drawn map" answer rather than pursuing AI-generated map art (which would fork into the same new-image-vendor problem as the Illustrated Storybook idea) — see `project_resonance_backlog_decisions.md` for the open question. Scoped to custom worlds only; Resonance and Manlandia keep their existing real maps untouched.

## Manual ability/magic toggles (player-driven, not GM-tagged)

Separate from the GM-narrated `[ABILITY ...]`/`[ABILITY N: used]` bracket tags above — these are the player's own "tap to mark used" buttons: Resonance's `.ability[data-ability]` buttons and Lyra's `#lyra-magic` charge button, plus Manlandia/custom's `.manlandia-ability-btn` (all wired in `public/game.js`, `setupAbilityToggles()`/the delegated `tab-characters` click handler). Every one of these confirms (a native `confirm()`, using the button's own visible label as the prompt text) before firing `toggle_ability`/`use_magic` on `POST /api/state` — added after a real accidental tap silently spent one of Lyra's Resonance charges with zero confirmation. A tap that's answered "no" is a complete no-op; the button's `used`/`available` state and the charge count are only ever mutated by the server's response.

A Manlandia/custom hero renders one `.manlandia-ability-btn` per power — the starting ability plus one more per unlocked bonus ability (see "Character growth" below) — each with its own independent used/available state. A bonus button carries `data-ability-id` (the starting button doesn't); the shared click handler uses that to route to `toggle_bonus_ability` (flips membership in `bonus_abilities_used`) instead of `toggle_ability` (flips the shared `ability_used` boolean) — same confirm-first pattern either way.

## Character growth (XP)

`lib/growth.js` — **Manlandia and custom campaigns only.** Deliberately excluded: Resonance's Lyra and Fen already have 3 fixed, bespoke abilities apiece from session 1 with no concept of an "unlockable pool," so the whole system just doesn't apply there. Stats (`ARCHETYPE_STATS`, in `lib/character-options.js` — the shared source of truth for archetype/ability ids, imported by `api/characters.js` and `lib/growth.js`'s `ABILITY_POOL` alike; each prompt file still keeps its own separate label/flavor-text map, deliberately not merged in) are also untouched by growth — set once at creation and never grow, since this system has no monster/difficulty scaling for a stat bump to stay balanced against.

- **XP sources**: a baseline amount (`GROWTH_CONFIG_KID`/`GROWTH_CONFIG_ADULT`'s `baselineXp`, currently 10) is awarded to every created hero (has an `archetype`) on every `new_session`. On top of that, the GM can award a bonus via `[XP N: +amount]` (N = player number) for a notable moment — parsed by `lib/gm-tags.js`'s `extractXpBonuses()`, applied through the same `applyStateTags()` path as every other tag, so it's automatically deferred while a roll is pending like everything else.
- **Two kinds of growth**, both driven by cumulative XP thresholds (`applyXpGain()` in `lib/growth.js`):
  - **Milestone badges** (`badgeXp`) — purely cosmetic flavor text (`FLAVOR_BADGES`, cycling through a fixed list) pushed to `character.milestones`, fired every time XP crosses another multiple of the threshold. No mechanical effect, no balance risk, frequent positive feedback.
  - **Ability unlocks** (`abilityXp`) — at a bigger, evenly-spaced threshold (threshold scales as `(unlocksSoFar + 1) * abilityXp`, so each successive unlock takes as much again), sets `character.pending_choice = { options: [...] }` offering up to 3 abilities from `ABILITY_POOL` the hero doesn't already have (their starting `ability_id` plus anything already in `bonus_abilities`). **Not auto-granted** — the player must resolve it via `POST /api/state` `action: "choose_ability"` (`{ character, ability_id }`), which calls `chooseAbility()` and adds the pick to `bonus_abilities`.
- **Kid vs. adult pacing**: `GROWTH_CONFIG_KID` (badges every 15 XP, ability unlocks every 50) applies to Manlandia and non-adult-flagged custom campaigns; `GROWTH_CONFIG_ADULT` (25 / 80) applies to adult-flagged custom campaigns — same `worldConfig.adult` check used for the harm auto-heal split in `new_session`.
- **Bonus ability used-tracking**: each unlocked bonus ability has its own once-per-session used/available state in `character.bonus_abilities_used` (an array of ability ids used this session), separate from the starting ability's `ability_used` boolean — added 2026-07-05, see the `[ABILITY N: used]` row above for how a bonus-ability use is told apart from the starting one. `charLine()` in both `lib/prompt-manlandia.js`/`lib/prompt-custom.js` shows each bonus ability's used/available state next to its name (`"Also learned: Lucky Break (available), Ancient Magic (used)"`), and the prompt's ABILITY USED instruction now asks the model to name which power was used whenever a hero has more than one. `new_session` resets `bonus_abilities_used` to `[]` alongside `ability_used`, same as every other once-per-session flag. Still no live transcript data on how reliably the model actually names the right ability when a hero has several — worth re-checking via `npm run check-drift` once real multi-ability transcripts exist.

## Hero customization + level-up moments (Manlandia/custom growth system only)

Two small additions riding alongside the growth system above, same Manlandia/custom-only scope (Resonance's Lyra/Fen are bespoke and don't have either) — **provisional**, per Matt's recommendation-accepted-pending-confirmation status, see `project_resonance_backlog_decisions.md`.

- **Customization**: `character.color` / `character.symbol`, a curated fixed color swatch (`HERO_COLORS`/`HERO_COLOR_IDS` in `lib/character-options.js`) and emoji list (`HERO_SYMBOLS`) — a free-form picker was deliberately rejected in favor of the same "fixed choice grid" pattern archetypes/abilities already use, which guarantees the color stays legible against the parchment theme and needs no new asset pipeline. Validated server-side in `api/characters.js` against those same lists; omitting either field on an edit preserves whatever was already saved, same pattern as `photo`. The wizard's step 4 (photo + backstory) gained two picker rows for these — no new wizard step. `public/game.js` duplicates the hex/emoji values client-side (`HERO_COLOR_HEX`) since this plain-script frontend has no import from `lib/` — kept in sync by hand, same accepted duplication as `ARCHETYPE_DISPLAY`/`ABILITY_DISPLAY`. Rendered as the character card's border color (`cardEl.style.borderColor`) and a small badge on the avatar's corner (`.char-avatar-symbol` — required changing `.char-avatar` from `overflow: hidden` to `overflow: visible` with the circular clip moved onto the `<img>` itself, since a corner-positioned badge was getting silently clipped by the old circular overflow mask).
- **Level-up moments**: a one-time in-place banner (`flashLevelUpBanner()`) plus a brief border pulse on the hero's own card, firing when a milestone badge count grows or a `pending_choice` first appears. Chosen over a full-screen takeover to match the app's existing "don't over-interrupt" feel (the same reasoning behind the tap-to-expand character details). There was no "did this just happen" signal anywhere before this — `renderManlandiaCard()` re-renders the full milestone list from scratch every poll — so `growthMomentState` (a small module-level map, same shape/purpose as the sound engine's tracker above) diffs the previous vs. current count purely to fire once. Pure CSS `@keyframes`, so it inherits the app's existing blanket `prefers-reduced-motion` rule for free.

## Solo/private scenes (Resonance only)

Lets one player mark a turn as a moment their partner's character "isn't in the room" for — dramatic irony (Lyra investigating alone, a secret doubt) that reveals in the log afterward as something the other player wasn't present for. Resonance-specific: its fixed two-character shape is what makes "my partner isn't in the room" a meaningful mechanic at all. **Not real security** — this is a UX device for two people who trust each other, not access control against a determined bypass. Matt's own framing, from when this was scoped: since he and Michelle sometimes share one device and sometimes use separate ones, the mechanic needed to work either way, which is what the PIN adds on top of simple entry-hiding.

- **Marking a turn private**: a "🎭 Private scene" toggle (`#private-scene-toggle`, `public/game.js`'s `setupPrivateScene()`) next to the message input, Resonance-only. While active, `POST /api/gm` is sent with `private: true`; `api/gm.js` stamps `private_to: player` on both the player's entry and the GM's response (and every subsequent roll_result in the same exchange, automatically — the flag is read fresh from the live `privateSceneActive` variable on each `sendToGM()` call, including its own internal roll_result recursion, so no extra plumbing was needed for that case). Turn notifications are suppressed for a private turn (`api/gm.js`) — the whole point is the other player doesn't know yet, and a push would tip them off to go looking for nothing.
- **Filtering**: `/api/poll` takes a `?player=<key>` query param identifying who's currently viewing, and excludes any entry whose `private_to` doesn't match (`e.private_to && e.private_to !== viewer`). No `player` param at all hides every private entry rather than showing everything — the safe default when nobody's identified as the viewer, and harmless for every world/entry that never sets `private_to` in the first place (i.e. everything except an active Resonance private scene). `public/game.js`'s `pollUrl()` always sends the live `currentPlayer` on every poll call.
- **Revealing**: manual only (no automatic reveal-at-next-turn or session-end) — `action: "reveal_scene"` (`{ character }`) on `POST /api/state` clears `private_to` from every entry belonging to that character, no LLM round-trip, same pattern as `end_combat`. The "👁 Reveal Scene" button (`#reveal-scene-btn`) only appears once the client has actually seen a `private_to === currentPlayer` entry come through (`setRevealSceneVisible()`/`hasPrivateScene` in `game.js`) — that flag only ever gets set true by a real sighting, never reset by an ordinary poll tick that happened to fetch nothing new (only a genuine full reload, or a successful reveal, resets it), since an incremental poll tick's absence of new private entries says nothing about whether older ones are still pending reveal.
- **The PIN, and why switching characters now reloads the log**: `characters.lyra.scene_pin` / `characters.fen.scene_pin` (optional 4-digit string, `action: "set_scene_pin"`) gates *switching* to that character via the Story tab's FEN/LYRA buttons — entered once per PIN per page load (`unlockedScenePins`, `public/game.js`), not persisted, re-prompted after a refresh. This is what makes the mechanic work on a **shared** device too, not just separate ones: without it, tapping the other character's button would trivially reveal whatever's private to them. Because filtering is keyed off `currentPlayer` (via `pollUrl()`), switching characters now **reloads the story log from scratch** (`completePlayerSwitch()` clears `logEntries`, resets `lastTimestamp` to 0, and re-runs `loadExistingLog()`) whenever the target is Lyra or Fen — necessary because a poll tick only fetches entries newer than `lastTimestamp`, so an older private entry that only just became visible under the new viewer would otherwise never be fetched at all. Manlandia/custom hero-switching is untouched — no PIN, no reload, since private scenes don't exist outside Resonance.
- **`scene_pin` is checked client-side and travels in plaintext** on ordinary `/api/gm`/`/api/poll` responses (a curious family member could find it via the browser's Network tab, not just guess it via a casual tap). A 2026-07-05 security review flagged this as worth a real fix — moving verification server-side — but Matt's call was that it's overkill for what this feature actually protects. Deliberately left as-is; don't "fix" this without checking with him first.
- **Visual marker**: a `.private` class + "🎭 Private Scene" / "🎭" label distinguishes a private entry from ordinary narration (`public/style.css`), visible only to whichever character can currently see it in the first place.
- **Scope note**: doesn't generalize to Manlandia/custom's multi-hero shape (which of N heroes is "absent" is a materially harder question) — explicitly out of scope, Resonance's fixed two-character structure only.

## Bonds (adult games only)

Dungeon World-style relationship statements between party members — e.g. "I trust Lyra with my life." Adult games only (Resonance always, adult-flagged custom campaigns) — kid games don't have this at all, an explicit decision (not an oversight) since the "everyone's aware of each other" table dynamic that motivates Bonds fit adult play better than kid play. Unlike every GM bracket tag above, **Bonds are entirely player-authored, not GM-narrated** — there's no `[BOND: ...]` tag for the model to emit. The player picks another party member and writes the statement themselves, so this goes through `POST /api/state` (same as `recover_harm`/`choose_ability`, other player-driven actions) rather than `/api/gm`.

- **Data shape**: `characters[key].bonds` — an array of `{ target, text, resolved }`, one array per adult character. Only exists on characters in worlds where Bonds apply — for custom campaigns this is decided once at creation from `worldConfig.adult` (`lib/gamestate-custom.js`), same trick `inventory` uses so neither a kid nor an adult custom campaign carries the other's permanently-unused field.
- **`action: "add_bond"`** (`{ character, target, text }`) — validates `character`/`target` both exist, `character !== target` (no self-bonds), and `text` is non-empty after trimming (capped at 200 chars). Pushes `{ target, text, resolved: false }`. Lazily creates the `bonds` array if missing, for adult campaigns created before this feature shipped.
- **`action: "resolve_bond"`** (`{ character, index }`) — marks that bond's `resolved` true. Idempotent: resolving an already-resolved bond is a no-op, not a double reward. **Reward differs by world**: adult custom campaigns (which do have the growth/XP system, see above) get a flat bonus via `applyXpGain(character, GROWTH_CONFIG_ADULT.bondXp, GROWTH_CONFIG_ADULT)` — currently 15 XP. Resonance has no growth system at all (see above), so resolving a bond there is purely cosmetic — no numeric reward, just the strikethrough in the UI. Both actions reject outright (400) if the world isn't adult at all (Manlandia, non-adult custom).
- **UI**: each adult character's card gets a "🔗 BONDS" section (`renderBondsSection()` in `public/game.js`) — a read-only list of existing bonds (with a "✓ Resolve" button on unresolved ones) plus a small add-bond form (target picker + text input). The add-bond form is deliberately built once and never rebuilt on subsequent renders, unlike every other render function in this file — rebuilding it on every poll tick would wipe out whatever the player is mid-typing. Resonance's picker is skipped entirely (only one other character exists, so the target is implied); custom worlds with more than 2 heroes get a real `<select>`.

## Remember-this pins (all worlds)

A player-authored escape hatch for the fact that the GM only ever sees the last `MAX_HISTORY` (40) `sessionLog` entries — universal across every world type (unlike Bonds above), since there's no reason a kid world's player is less entitled to flag "don't lose this" than an adult one. Like Bonds, **entirely player-driven, not GM-narrated** — no bracket tag involved, no model-compliance risk. A 📌 button sits in the header of every rendered story-log entry (both the GM's own narration and each player's line, via `appendGMEntry()`/`appendPlayerEntry()` in `public/game.js`), and pins that entry's own text verbatim — this deliberately covers "a player wants to remember something the GM said" as well as their own line, without needing the player to retype it.

- **Data shape**: `worldState.pinned_notes` — a flat array of `{ text, timestamp }`, shared party-wide (same visibility model as NPCs/Objectives/inventory, not per-player). Capped at the 10 most recent; `action: "add_pinned_note"` (`api/state.js`) drops the oldest once the cap is exceeded, the same "small and bounded, not a second session log" reasoning as `author_note`'s 1000-char cap.
- **Prompt injection**: fed into all three prompt files (`lib/prompt.js`, `lib/prompt-manlandia.js`, `lib/prompt-custom.js`) as a `REMEMBERED MOMENTS` block, positioned right after `AUTHOR'S NOTE` and using the same never-contradict framing — the only difference between the two blocks is who curates them (a parent, ahead of time, vs. a player, mid-scene).
- **UI**: read-only, rendered inside the existing Author's Note panel in the Settings tab (`renderPinnedNotes()` in `public/game.js`) rather than a new panel of its own — fetched fresh via `GET /api/state` every time that panel opens (unlike the Author's Note textarea itself, which is only ever prefilled once per page load, so a pin added mid-session can't clobber a parent's in-progress unsaved edit to the note).
- **Not build here**: no "promote a pin into the permanent Author's Note" action, and no per-player-private visibility — both considered and explicitly deferred as unnecessary complexity for a first pass.

## Recap

`GET /api/recap` (`api/recap.js`, `lib/recap.js`) summarizes the transcript for whoever's picking the story back up. Before 2026-07-19 this wrote the recap in second person from one specific character's POV (`?player=<key>` → `getPlayerDisplayName()` → "you" = that character, others named/"they") — added so different players wouldn't get an arbitrarily-fixed POV (originally every player got the same recap written from whoever happened to be `currentPlayer` first, confirmed live: both Lyra's and Fen's recaps came back written from Fen's POV). That fix addressed the wrong half of the problem: it made the POV *correct per viewer* but still POV-*locked to one character*, so the recap only read naturally for whoever it was generated for — not something any hero's player could pick up and have make sense to them. `buildRecapSystemPrompt(isKidWorld)` now always writes in **third person, naming every character by name** ("Fen crept through the tunnel while Lyra kept watch from the rooftop") and explicitly never addresses the reader as "you" — this makes the same recap text correct for any player at the table, so the `?player=` query param and the whole viewer-POV branch were removed entirely rather than kept as a now-redundant option.

**Manual trigger**: the "📜 Recap" button (`#recap-btn`, Story tab) calls `loadRecap()` any time, no threshold — always available regardless of how recently the player was last here.

**Share**: a "📤 Share" button (`#recap-share-btn`) in the recap overlay's action row calls `shareRecap()`, for sending the recap to someone outside the app (a grandparent, an absent player) as a highlights digest. Tries the native Web Share API (`navigator.share`) first — nicer on phones, and this app is phone-first — falling back to a clipboard copy (`navigator.clipboard.writeText`) everywhere else, including a cancelled/unsupported share sheet. `#recap-share-status` shows a transient "Copied to clipboard!" (or a failure message if even the clipboard write fails) for 3s after a fallback copy; the native share path shows no status at all since the OS's own share sheet is the confirmation.

**Read aloud**: a "🔊 Listen" button (`#recap-speak-btn`) in the recap overlay's action row, added 2026-07-19, reuses the same `speakText()`/`stopSpeech()` machinery as every per-entry 🔊 button in the Story log — same voice selection, same pause/resume iOS keepalive, same toggle-to-stop-on-second-click behavior. The one wrinkle: per-entry speak buttons are icon-only, but this one carries a label ("🔊 Listen"), and the shared `speakText()`/`stopSpeech()`/`finishSpeech()` used to just overwrite `btn.textContent` wholesale — fine for an icon-only button, but it would silently eat the " Listen" label the first time it was clicked. Fixed by routing all three through a shared `setSpeakBtnIcon(btn, icon, speaking)` helper that reads an optional `data-speak-label` attribute (set on `#recap-speak-btn`, absent on every per-entry button) and only appends it back if present — per-entry buttons are byte-for-byte unaffected. Closing the recap overlay (`#recap-close-btn`) calls `stopSpeech()` so audio doesn't keep playing after the overlay's gone, and reopening/reloading the recap (`loadRecap()`) does the same before showing "Loading recap…", so a stale utterance can't talk over new text.

**Auto-surfaced catch-up banner** (2026-07-19): `checkCatchUp()` in `public/game.js` proactively nudges a returning player toward the same recap instead of relying on them to remember the button exists. Tracked via `localStorage` key `last_seen_<world>_<player>`, stamped with `Date.now()` every time `checkCatchUp()` runs (initial load, world switch, character switch); if the *previous* stamp is more than `CATCHUP_THRESHOLD_MS` (24h) old, a dismissible `#catchup-banner` appears in the Story tab ("👋 Welcome back — it's been N days...") with a "View catch-up" button (opens the same recap overlay) and a "✕" dismiss (just hides the banner, doesn't suppress future ones). **Purely client-side and per-device, not per-player globally** — there's no server-side "last seen" field, so this is the same trust model as `adult_pin`/`unlockedScenePins` elsewhere in this file: a player who alternates between a phone and a laptop gets independent "last seen" tracking on each, and could see the banner on a device even the same day they played on another. Considered and rejected: a server-side per-character `last_seen_at` field, which would fix the cross-device gap but adds a new `worldState`/`characters` field and a write on every turn for a "nice to have" nudge — revisit only if the device-local approximation proves actually annoying in practice. A first-ever visit on a device (no prior stamp) never shows the banner, which also means a transcript-less brand-new world never triggers it — no special-casing needed against `/api/recap`'s own "hasn't begun yet" fallback text.

## Session log mechanics

- `gameState.sessionLog` is the running turn-by-turn history. `api/gm.js` sends only the most recent `MAX_HISTORY` (40) entries to the LLM as context, to bound prompt cost — the full log is still stored.
- Once `sessionLog.length > 100`, it's trimmed to the most recent 80 entries (keeps the Redis payload bounded).
- **`type: "begin"` is idempotent as of 2026-07-06.** The one-time opening narration used to have nothing stopping a second submission from generating a brand-new, unrelated opening scene — confirmed live in a 2026-07-05 playtest: three separate `[SESSION BEGINS]` calls (a device race, each thinking it was first) landed in one campaign's `sessionLog`, one narrating characters that were never actually created. `api/gm.js` now checks `gameState.sessionLog.length > 0` before calling Groq at all for a `begin` submission and, if the log already has content, returns `{ alreadyBegun: true, gameState }` as a true no-op — not an error, since a duplicate begin from a race isn't a client bug worth surfacing, unlike a stale `roll_result`. `public/game.js`'s `sendToGM()` handles this the same way it already handles `data.waiting`: resync cached state, append nothing.
- **Roll flow**: when the GM calls for a roll, both the triggering user entry and the GM's response are pushed with `.rolling = true` and are excluded from `/api/poll` results. The dice roll itself is fully client-automatic — `public/game.js`'s `sendToGM()` sees `needsRoll: true` in the direct response, animates the dice, and immediately recursively calls itself with `type: "roll_result"` in the same call chain. There's no manual "roll" button and no separate confirmation step. When that `roll_result` call comes in, the flagged entries are un-flagged and their timestamps pushed back to `Date.now() - 2` (so they still sort before the new entries being added in that same call) — this is what keeps a roll's "before" and "after" entries appearing in the right order without a real multi-turn transaction.
  - **State tags are deferred while a roll is pending.** `lib/apply-state-tags.js`'s `applyStateTags()` (LOCATION, SCAR, CURSE, CHARACTER harm, ABILITY used, etc.) is only called when a turn is *not* requesting a roll. A roll-request response's tags are left sitting in the stored (but hidden) entry content and applied retroactively once `roll_result` comes in, by re-running `applyStateTags()` against that stored content. This exists because a real live Manlandia turn got interrupted mid-roll (client closed/dropped before the automatic `roll_result` call completed) and the tags had already applied — `worldState.location` had silently advanced to a location the player never saw narrated, since `/api/poll`'s `worldStatePayload` reads live `worldState` with no awareness of the `rolling` flag on the hidden entries that caused it.
  - **Recovering an interrupted roll**: since there's no manual retry path, `/api/poll` also returns `pendingRoll: { stat, advantage, player } | null` whenever the *very last* sessionLog entry is a still-rolling GM entry carrying a persisted `rollStat` (only ever true for entries created after this fix — the `rollStat`/`rollAdvantage` fields didn't exist before). `game.js`'s `resumePendingRollIfAny()` checks this on initial load and on every poll tick, and if it matches the current player, replays the animate-roll → submit-roll_result chain automatically. It only looks at the *very last* entry, so if a newer turn has since been added the game has already moved past the stuck roll and it's correctly left alone. Deliberately does **not** retroactively surface an old stuck entry that predates the `rollStat` field (the gate excludes it) — a real one was sitting in Manlandia's live log as of 2026-07, checked again 2026-07-05 via `GET /api/state` and no longer present (see `project_resonance_backlog_decisions.md`'s Resolved section), most likely aged out naturally as new turns were played. `pendingRoll.player` reads `lastEntry.rollPlayer` (see "Wait-for-all-players turn gating" below) first, falling back to the preceding user entry's own `.player` for anything saved before that field existed.
  - **Duplicate roll_result protection**: this same recovery path is exactly what let a real "double roll" bug happen — if the same player has the game open on two devices (a phone and a laptop, or a shared login), both independently see the same `pendingRoll` and can both fire `resumePendingRollIfAny()` for it, each submitting its own `roll_result` and each getting a full GM response, producing two duplicate narrations for one roll. Two fixes in `api/gm.js`, together closing both the concurrent and the delayed version of this race: (1) `type: "roll_result"` no longer skips the per-world `gmlock` — every submission type takes it now, so two requests arriving at the same instant are serialized rather than both reaching Groq; (2) a `roll_result` is rejected outright (`409`) unless a matching `role: "user", rolling: true` entry for that player still exists in `sessionLog` — this is what catches a *delayed* duplicate that arrives well after the first one already resolved and released the lock.
- **New session** (`api/state.js`, `action: "new_session"`): archives the current `sessionLog` + a player-written summary into `worldState.session_archive`, increments `session`, clears `sessionLog` and `worldState.pending_turn`, and resets per-world ability flags (Resonance: `lyra`/`fen`'s once-per-session abilities + `magic_uses_remaining`; Manlandia/custom: every `playerN.ability_used` and `playerN.bonus_abilities_used`).

## Authorization: game secret vs. adult PIN

Two independent, orthogonal checks, both simple shared-secret headers (no sessions, no tokens):

- **`X-Game-Secret`** — required on state-mutating POSTs (`api/gm.js`, `api/state.js`, `api/characters.js`, `api/campaigns.js`, `api/push.js`). Fails **open** if `GAME_SECRET` is unset (`if (gameSecret && header !== gameSecret)`) — this is existing, long-standing behavior; don't "fix" it without checking whether anything relies on local/no-env-var dev use.
- **`X-Adult-Pin`** — required on any request that touches an adult-gated world: Resonance always, or a custom campaign with `worldConfig.adult === true`. Enforced by `lib/adultgate.js`'s `checkAdultAccess(req, res, worldConfig, gameState)` — **`async`**, must be `await`ed — called after each handler loads state (so no extra Redis round-trip except in `api/state.js`'s POST branch, which loads state early specifically to check this). Wired into `api/gm.js`, `api/poll.js`, `api/recap.js`, `api/state.js`, `api/help.js` — every endpoint that can read or narrate world content. Unlike the game secret, this **fails closed**: if `ADULT_PIN` isn't configured, adult worlds stay locked rather than opening up. The client stores the raw PIN in `localStorage` (`adult_pin`) after a successful `/api/unlock` call and resends it on every request via `authPost()`/`authGet()` in `game.js` — same trust model as `X-Game-Secret`, just a second independent secret for a second boundary (family-wide vs. kids-vs-adult-content). The underlying `isAdultWorld(worldConfig, gameState)` check `checkAdultAccess` calls internally is also exported and reused directly wherever *game logic* (not access control) needs the same "is this world adult" boolean — `api/state.js` (Bonds, `new_session`'s harm-heal split), `api/recap.js` (kid-safe vs. adult recap tone), `lib/apply-state-tags.js` (bonus-XP config). One helper, one definition, instead of each call site re-deriving the same expression by hand.
  - **Rate-limited on wrong guesses, 2026-07-05**: a 4-digit PIN is only 10,000 combinations, and `checkAdultAccess()` runs on every one of those five endpoints — `GET /api/poll` especially was a cheap guessing oracle, since it needs no `X-Game-Secret` and is legitimately hit every ~8s by every connected client. `checkAdultAccess()` now calls `checkRateLimit()` (same Redis INCR+EXPIRE limiter `/api/unlock` and `/api/help`/`/api/recap` already use) **only on a wrong or missing pin** — 5 wrong attempts per 60s per IP, then 429 instead of 403. Counting failures only, not every call, means a device that already has the correct pin is never throttled no matter how often it polls; only an actual guess counts against the limit, so real usage is completely unaffected. Same 5/60s bar as `/api/unlock`'s own limit, for one consistent answer everywhere the pin is checked.
- `api/campaigns.js` GET (the world-selector listing) and `api/unlock.js` itself are deliberately **not** gated — you need to see a campaign exists (name/theme/adult flag) before you could ever supply the right PIN for it, and unlock is how you obtain the PIN check in the first place.

## Rate limiting

`lib/ratelimit.js`'s `checkRateLimit(key, limit, windowSeconds)` is a simple Redis `INCR`+`EXPIRE` fixed-window counter, used in two contexts with two different limits:
- **Paid-API cost control**: `api/help.js` and `api/recap.js` (10 req/60s per IP) — the two endpoints that call the paid Groq API with **no** `X-Game-Secret` required (both intentionally auth-free so kids can ask for help without a login). Generous for real use, tight enough to stop a loop from running up Groq costs. `api/gm.js` isn't rate-limited by IP since it already requires `X-Game-Secret` — it has its own mechanism below instead.
- **PIN-guessing protection**: `api/unlock.js` and `lib/adultgate.js`'s `checkAdultAccess()`, both 5 *wrong-guess* attempts/60s per IP, counting failures only — a correct pin is checked and returned first, before the rate limiter is ever touched. This matters beyond just "don't punish normal use": Matt's whole family shares one home IP, so counting every attempt (including correct ones, `api/unlock.js`'s original version) could let several people unlocking around the same time lock out the rest of the household — found live during testing when repeated correct-PIN calls from the same script/IP briefly 429'd. A 4-digit `ADULT_PIN` is only 10,000 combinations, and before this existed, any of the five `X-Adult-Pin`-gated endpoints (especially `GET /api/poll`, hit every ~8s with no secret required at all) was a cheap brute-force oracle.

`api/gm.js` also caps the `message` field at 1000 characters (mirrors the 500-char cap `api/help.js` already had on `question`) — there was previously no limit on the one field that reaches the LLM with no size check at all.

**Groq's own rate limit** (a real quota shared across the whole account — every world draws from one `GROQ_API_KEY`) is handled two ways:
- `lib/gemini.js` retries once, after a 1.5s delay, on a 429 from Groq before giving up — absorbs most transient spikes silently. If that retry also 429s, `generateContent()` falls back to `llama-3.1-8b-instant` (a separate quota bucket from the primary 70b model, confirmed live) with a trimmed 6-entry history and a 400-token response cap — a real quota exhaustion still degrades narrative quality rather than hard-failing the turn. If the fallback itself fails for any reason, the original primary 429 (with its real `retryAfterSeconds`) is what surfaces to the player, not a confusing fallback error.
- If it still fails, `api/gm.js` returns a 429 with player-facing copy instead of leaking Groq's raw quota-error text. `lib/gemini.js` extracts how long Groq actually wants us to wait (`err.retryAfterSeconds`, read from the `Retry-After` header first, falling back to parsing "try again in X.Xs" out of the error message itself) and `api/gm.js`'s `formatWaitMessage()` includes that in the copy ("wait about 18 seconds") whenever it's known and ≥5s — below that threshold (or when neither source is available) it just says "wait a few seconds", since a precise sub-5s number doesn't help anyone.
- Separately, `api/gm.js` holds a short-lived Redis lock (`gmlock:<worldId>`, `SET NX PX 20000`, released as soon as the Groq call resolves) around every non-`roll_result` call, so two players in the *same* world submitting at literally the same instant don't both hit Groq concurrently — the second gets a 429 ("Another turn just came in for this world") instead. This is a true in-flight lock, not a fixed cooldown: a solo player whose turn resolves quickly can submit their very next action immediately, since the lock is already released by then. It's scoped per-world, not global, so different family members playing different campaigns at the same time are never affected by each other.
- On any error response from `/api/gm`, `public/game.js`'s `submitAction()` puts the player's typed message back in the input box (via `sendToGM()`'s boolean return) so they can just hit send again instead of retyping.

**Tracking how often Groq's free-tier quota actually gets hit** (2026-07-19): added after a family playtest hit the free tier's per-minute token limit despite feeling like "very little" play — measuring the real system prompt + a full 40-entry history against the free tier's 12K TPM cap put the realistic sustainable rate around **2 GM turns/minute site-wide** (not per-world — one shared `GROQ_API_KEY`), tightening as a session's history fills toward `MAX_HISTORY`. Rather than guess whether that's a one-off or a recurring problem, `lib/groq-tracking.js`'s `recordRateLimitHit(worldId)` is called from both `api/gm.js`'s and `api/recap.js`'s catch blocks — the two endpoints that call Groq directly — only on the *player-visible* failure path (`err.status === 429` after `api/gm.js`'s retry **and** fallback model in `lib/gemini.js` have both already failed; `api/recap.js` has no retry/fallback of its own, so any 429 there is immediately player-visible). Not called for every transient blip that self-resolves, since those aren't what the player experienced as "hitting the limit." (Caught a real gap here during the 2026-07-19 closeout: the first version only instrumented `api/gm.js`, undercounting real hits — a live recap call failed on the exact same day from the **daily** token cap, a separate dimension from the per-minute one, being at 99,066/100,000 used.) Writes `{ts, world}` to a capped Redis list (`groq:ratelimit:events`, newest-first, trimmed to the most recent 500) plus an uncapped `groq:ratelimit:total_count` counter, both best-effort (wrapped in try/catch — a tracking write failing must never affect the real error response already being sent). `getRateLimitStats()` reads both back and aggregates total/last-24h/last-7d/most-recent/by-world.

Exposed two ways: `GET /api/groq-stats` (`api/groq-stats.js`, same fail-open `X-Game-Secret` gate as every other endpoint) for programmatic checks, and `scripts/check-groq-ratelimit.js` (same "live-network, not part of `npm test`" convention as `scripts/check-tag-drift.js`) for a human-readable summary — `GAME_SECRET=... node scripts/check-groq-ratelimit.js`. Matt chose to stay on the free tier for now rather than upgrade preemptively; this is explicitly **the thing to check before that conversation comes up again** — run the script (or hit the endpoint) and bring real frequency data, not another estimate, before recommending whether to upgrade to Groq's paid Developer tier.

## Sound effects

`public/game.js` has a small synthesized-tone audio engine (`playTone()`, built on `AudioContext`/`OscillatorNode`) instead of any audio asset files — keeps the zero-new-dependency, zero-asset-pipeline policy intact. Gated behind a `#sound-fx-toggle` in the Settings tab (`soundFxEnabled`, `localStorage`-persisted as `sound_fx`, **off by default** — same opt-in convention as the existing Audio/auto-read toggle). `getAudioCtx()` returns `null` whenever sound is off, so every `playX()` call is a safe no-op rather than needing an `if (soundFxEnabled)` guard at every call site. Background/mood music (matching the Story Mood tone slider) is explicitly out of scope — a real looping/crossfade problem, not a quick addition.

Hooked into three moments: `animateRoll()`'s shuffle tick (`playDiceTick()`, once per shuffle frame) and its result stinger (`playRollResultSound(level)` — a distinct short phrase per success/partial/failure/disaster band, disaster using a lower `sawtooth` buzz instead of the others' brighter `triangle` tone); and Manlandia/custom hero growth (`playBadgeChime()`/`playLevelUpChime()`, Resonance excluded since it has no growth system at all). Growth has no "did this just happen" signal anywhere else in the app — `renderManlandiaCard()` re-renders the full milestone list from scratch every poll — so `checkGrowthMoment()`'s small module-level `growthMomentState` map (shared with the level-up banner below, see "Hero customization + level-up moments") tracks each hero's last-seen milestone count/`pending_choice` state purely to detect a same-session increase and fire once, without touching how the badges themselves render or persisting anything server-side.

## Responsive width + wide-screen story sidebar

`#app`'s width was a fixed 680px until 2026-07-19; it's now `max-width: var(--content-max)` where `--content-max: clamp(680px, 94vw, 1400px)` (`public/style.css`). On phones this is a no-op — the clamp floor never forces the layout wider than the actual viewport — but on tablets/desktops the app now uses more of the available width instead of sitting in a fixed narrow column with large empty margins on both sides. The two bottom-sheet overlays that visually match `#app`'s width (`.help-box`, `.end-session-box`) share the same variable; `.recap-box` (a separate centered modal, not a bottom sheet) gets a smaller matching clamp, `clamp(420px, 90vw, 720px)`, so it widens proportionally without becoming as wide as the main app frame.

Simply raising the cap has a ceiling, though: past a point, a wider `#app` just means longer narration lines (already ~150+ characters at 1100px, uncomfortably long for prose), not less scrolling. So at `min-width: 1100px`, `#tab-story` switches from its normal single-column layout to a row: the tab's existing content (log, input, banners) moved into a new `#story-main` wrapper, sitting next to a new `<aside id="story-sidebar">` (width `clamp(280px, 22vw, 380px)`) that surfaces state a player would otherwise have to switch tabs to see — party harm status, current location, and up to 5 open Objectives/Leads (merged into one condensed list, with a "+N more — see Map tab" note if truncated). This is what the raised `--content-max` cap is actually for: past this breakpoint, extra width goes to the sidebar instead of only ever lengthening narration lines further. Below 1100px, `#story-sidebar` is `display:none` and the layout is byte-for-byte the original single-column shape.

`public/game.js`'s `buildStorySummaryHTML(ws, characters)` builds this content (party rows, location, threads list) from data that was already flowing through the app — `getRealCharacterKeys()`/`getPlayerDisplayName()` (`public/pure.js`, same "real character" definition the waiting-on banner uses) for the party list, and `worldState.objectives`/`.clues` (already part of every `/api/poll`/`/api/gm` response via `buildWorldStatePayload()`) for the open-threads list — so no new API surface was needed. `renderStorySidebar()` calls it once and writes the same HTML into both `#story-sidebar` and the mobile summary panel below, so the two can never drift out of sync with each other. It's called at the end of `updateCharacterUI()`, the same function every other live-updating panel (harm, combat tracker, waiting banner) already goes through, so both stay in sync for free on every poll tick, turn, and initial load rather than needing separate wiring. Deliberately its own small set of `.sidebar-harm-chip.<Level>` color classes rather than reusing `.harm-value` directly — that class is also the clickable "tap to recover" control on the Characters tab, and its `:not(.Unhurt)` cursor/hover rules would follow along and make this read-only chip look clickable too.

**Narrow-screen counterpart**: below the sidebar's 1100px breakpoint there's no spare width to show this permanently without pushing the log down further, so a collapsible version sits in `#recap-bar` instead — a `#story-summary-toggle` button (`📍 <location> ▾`, next to the existing Recap button) that expands `#story-summary-panel` in place on tap (`setupStorySummaryToggle()`, a plain class toggle + `aria-expanded`, no overlay/modal). Collapsed by default so a phone's Story tab looks exactly like it always did until a player actually wants this info. Both `#story-summary-toggle` and `#story-summary-panel` are force-hidden (`display:none !important`) at the same `min-width: 1100px` breakpoint where `#story-sidebar` takes over, so the two never show at once.

All three rounds shipped the same day (2026-07-21): the raised max-width alone, then the wide-screen sidebar after Matt found the first pass still left large empty gutters on a genuinely wide monitor, then this collapsible mobile counterpart after he asked for the same info reachable on a phone without disturbing the existing compact layout there. See `project_resonance_backlog_decisions.md`'s Resolved section for the full history and reasoning.

## Action input box

`#action-input` is a `<textarea rows="1">` (changed from a plain `<input>` 2026-07-19), auto-growing as the player types or speaks so a multi-line action stays fully visible instead of scrolling its own contents sideways out of view. `autoResizeActionInput()` resets `style.height` to `"auto"` then sets it to `scrollHeight` (capped at `ACTION_INPUT_MAX_HEIGHT`, 160px — past that it scrolls internally rather than taking over the screen), wired to the box's own `"input"` event. `#action-row` switched from `align-items: center` to `flex-end` so the mic/send buttons stay pinned to the bottom as the box grows upward, chat-app style. Converting to a `<textarea>` also means the existing `!e.shiftKey` check in the Enter-to-send keydown handler now does something it couldn't before: **Shift+Enter inserts a real newline** instead of being a no-op, for free.

Setting `.value` programmatically (voice results, suggestion-chip clicks, restoring text after a failed send) doesn't fire an `"input"` event, so it wouldn't trigger the resize on its own — every such call site goes through `setActionInputValue(text)` instead of assigning `.value` directly, which sets the value and then calls `autoResizeActionInput()` itself.

**Draft text doesn't cross worlds** (2026-07-19 fix): `switchWorld()`'s already-initialized branch now calls `setActionInputValue("")` and `hideSuggestionChips()` alongside its existing per-switch resets (log, `lastTimestamp`, `cachedGameState`) — before this, `actionInput` was a single shared DOM element with no per-world state of its own, so a half-typed draft in one world would still be sitting in the box after switching to a different one (confirmed live: a player saw the same in-progress text appear in an unrelated campaign). Switching *characters* within the same world deliberately doesn't clear it — a draft is still relevant to the same story, just possibly a different speaker.

**Optimistic entry rollback on a failed send** (2026-07-19 fix): `submitAction()` appends the player's line to the story log immediately (`appendPlayerEntry()`, before `sendToGM()`'s network round-trip even resolves), so the UI feels instant. On failure — `sendToGM()` returning `false` (a Groq error, rate limit, dropped connection, etc.) — the existing behavior already restored the typed text into the box so the player could resend without retyping, but left the optimistically-appended log line in place. That produced a genuinely confusing, self-contradicting state live: the player's words visible in the story (implying the action went through) sitting right next to the same words back in the input box (implying it didn't). `appendPlayerEntry()` now returns the DOM node it created, and `submitAction()` calls `.remove()` on it in the failure branch, alongside restoring the text — so a failed send now reads as a clean "didn't happen yet, try again," not a half-succeeded one. Known residual gap: if the *first* leg of a roll-triggered action succeeds but the recursive `roll_result` leg then fails, only the original action's optimistic line is rolled back — the separate roll-result entry added in between (`appendRollResult()`) is left in place. Rare edge case (two round-trips have to succeed-then-fail in sequence); not fixed here, revisit if it shows up live.

## Voice input

`setupVoice()` wires `#voice-btn`/`#action-input` to the Web Speech API's `SpeechRecognition`/`webkitSpeechRecognition`. iOS Safari has supported this (prefixed) since Safari 14.5 — by 2026 effectively every iPad in use — so the mic button reliably renders there; a browser that genuinely lacks the API at all (the `if (!SpeechRecognition)` branch) hides the button rather than showing one that can't work.

**Errors used to fail completely silently** (2026-07-19 fix) — `recognition.onerror` just called `stopListening()`, which quietly re-hid the "Listening…" status with no explanation. A parent reported a player's iPad mic "not working" with no further detail to go on, which is exactly what this looks like from the outside: mic permission denied (or previously denied and never re-prompted), no network for Apple's/Google's server-side recognition, or any other `onerror` reason — all silently swallowed. `showVoiceError(message)` now sets `#voice-status`'s text to something the player can actually act on (a specific "microphone access is blocked" message for `not-allowed`/`service-not-allowed`, a generic "try typing instead" for anything else) and auto-hides it after 4s — `"no-speech"`/`"aborted"` are left as plain silent stops, since those just mean "didn't catch anything that time," not a real failure worth explaining. One subtlety: browsers fire `onend` right after `onerror` for the same session, which would otherwise immediately re-hide the message `showVoiceError()` just showed — `onend`'s handler now only calls the plain `stopListening()` (which does hide the status) when `isListening` is still `true`, i.e. nothing already handled this session's end.

## Push notifications

Notifies the other player(s) in a world when someone takes a turn — the one item that was in the backlog long enough to have its own "large effort, defer until needed" note.

- **Storage**: `push:<worldId>:subscriptions` in Redis — an array of `{ player, endpoint, keys: { p256dh, auth } }`. Multiple devices per player are fine; `api/push.js`'s `subscribe` action dedupes by `endpoint`.
- **Client**: `public/sw.js` (service worker, registered by `setupPushNotifications()` in `game.js`) handles the `push` and `notificationclick` events. The header's "Alert" button (`#notify-btn`, hidden by default, shown once the service worker registers successfully) drives `Notification.requestPermission()` → `pushManager.subscribe()` → `POST /api/push`. The VAPID public key needed for `subscribe()` is fetched from `GET /api/vapid-public-key` rather than hardcoded client-side, so it stays a server-configured value like everything else.
- **Sending**: `api/gm.js`'s `sendTurnNotifications()`, called after `setState()`, **only when the turn is fully resolved** (`!rolling` — the same flag that already hides in-progress roll entries from `/api/poll`, so nobody gets notified about content they can't see yet). Excludes the sender's own devices (`lib/push.js`'s `selectNotifyTargets()`). Notification text is deliberately generic ("X took a turn — tap to see what happens!") — never narration content, so there's no need to special-case the adult gate for what shows up in a lock-screen preview. Entirely best-effort: wrapped in try/catch, a push failure never breaks the GM response itself. A subscription that comes back 404/410 (the push service's way of saying "this one's dead") is removed from storage automatically.
- **iOS caveat**: Apple only allows Web Push for a home-screen-installed PWA, not a regular Safari tab (since iOS 16.4). `public/manifest.json` + the `apple-touch-icon`/`manifest` links in `index.html`'s `<head>` make that installable, but there's no way to make Safari-tab notifications work on iPhone — this is a real platform limitation, documented for players in `README.md`.
- **What can't be verified by tests**: everything up to "the correctly-addressed, correctly-encrypted request was handed to the push service" is covered (`tests/push.test.js`, `tests/api-push.test.js`, `tests/api-vapid.test.js`, the push-specific cases in `tests/api-gm-push.test.js`). Whether a real device actually *receives* it requires a real phone — there's no way to simulate a round trip through Apple's/Google's actual push infrastructure in an automated test.

## Waiting-on banner and turn-stall reminders

Turn *order* isn't enforced anywhere in this app — anyone can act anytime, in any order. Both features below are purely informational nudges on top of that, not a real turn-order system, built from a single `worldState.last_actor` field rather than tracking a real queue. This is a different question from whether the *story advances* on a single action alone — see "Wait-for-all-players turn gating" below, which is a real gate, not just informational.

- **`worldState.last_actor` / `last_action_at`**: set on every `POST /api/gm` submission (`api/gm.js`), regardless of roll state — the point is "did someone engage," not "did their turn fully resolve." Exposed to the client via both `/api/gm`'s response and `/api/poll`'s `worldStatePayload`.
- **`getRealCharacterKeys(world, characters)` / `getWaitingOn(lastActor, world, characters)`** (`public/pure.js`) — shared between the client and the server (the cron job below `require()`s `public/pure.js` directly, same dual-use pattern the rest of that file already relies on for testability). "Real" means an actual created hero (Resonance's Lyra/Fen always count; a Manlandia/custom player slot only counts once it has an `archetype`). "Waiting on" is just "every real character except whoever went last" — recomputed fresh each time, not a persisted queue.
- **Waiting-on banner**: `#waiting-banner` in the Story tab, rendered by `renderWaitingBanner()` inside `updateCharacterUI()` — so it refreshes everywhere that function already gets called (initial load, poll tick, right after a turn resolves) with no extra wiring. Hidden entirely for a solo game (nobody else to wait on).
- **Turn-stall push reminder**: `api/cron-turn-reminder.js`, triggered daily by Vercel Cron (`vercel.json`'s `crons` array, `0 15 * * *`). Checks Resonance, Manlandia, and every non-archived custom campaign from `campaigns:index`; if a world's `last_action_at` is more than 48h stale, sends a push (via the same `lib/push.js`/`web-push` plumbing `api/gm.js` already uses) to everyone subscribed except the last actor's own devices. Fires once per stall, not once per day forever — gated on `worldState.last_reminder_sent_at` being older than `last_action_at`, which a fresh action naturally invalidates. Skips a world outright if there's nobody real to remind (solo campaigns) or push isn't configured yet (no VAPID keys). A single world's failure (via try/catch per world) can't take the whole cron run down.
- **`CRON_SECRET`** (optional): if set, the cron endpoint requires `Authorization: Bearer <CRON_SECRET>`; if unset, it fails open — same convention as `GAME_SECRET`, and the blast radius of skipping this is "an extra push notification," not a data-safety risk. Vercel's own Cron feature calls this endpoint on its schedule regardless of whether the secret is configured; setting one just stops anyone else who finds the URL from triggering it on demand.

## Wait-for-all-players turn gating (merged joint turns)

Before this existed, a single player's submission to `POST /api/gm` advanced the story immediately, every time — the other real character's player could come back to find several exchanges had already happened without them (confirmed live in Resonance: whoever typed fastest kept moving the scene forward regardless of whether their partner had also weighed in). Now, in any world with more than one *real* character (`getRealCharacterKeys` — same "real" definition the waiting-on banner above uses), an action is held until every real character has submitted one for the round, then all of them are merged into a single joint turn.

- **`worldState.pending_turn`**: a plain object, `{ [playerKey]: { message, timestamp } }`, holding whoever has acted so far this round. `api/gm.js` adds the submitting player's entry on every `type: "action"` request (never `"begin"`, the one-time opening narration, and never `"roll_result"`, the second half of a turn already in flight — neither is ever held). If any real character hasn't submitted yet, the handler returns `{ waiting: true, waitingOn: [...] }` immediately — no Groq call, nothing pushed to `sessionLog` yet — and saves the partial `pending_turn` back to Redis so the other player's next poll sees it. Re-submitting overwrites your own prior pending entry (changing your mind before the other player has gone is a normal overwrite, not an error). Exposed to the client via `buildWorldStatePayload()` (shared by `/api/gm`'s response and `/api/poll`), same pattern as every other `worldState` field.
- **Merging**: once every real character's entry is present, `api/gm.js` joins them into one user message — `"Fen: I wipe down the bar\nLyra: I read my book quietly"` — and calls the GM exactly once. The resulting `sessionLog` still gets one `role: "user"` entry per contributor (so each renders as its own line in the story log, exactly like before), plus the one shared GM entry — `pending_turn` is cleared back to `{}` in the same write. `new_session` (`api/state.js`) also clears `pending_turn`, so a held-but-never-completed action from an ending session can't bleed into the next one.
- **Roll attribution for a merged turn**: see the `ROLL:STAT` row in the bracket-tag table above — the GM can now name which contributor is rolling (`ROLL:FEN:AGILITY`), since the device that happens to complete the merge and receive the real response isn't necessarily the character the roll is for. `api/gm.js` computes `roller = rollPlayer || player` and stamps it on the GM entry as `gmEntry.rollPlayer`, which `api/poll.js`'s `pendingRoll.player` and the final JSON response's `rollPlayer` field both read from — `public/game.js`'s `sendToGM()` uses `data.rollPlayer || player` to decide whose stats/name to use for the dice animation, instead of assuming it's always the submitting device's own character.
- **Notifications**: `sendTurnNotifications()` now excludes *every* contributor of a merged turn from the push, not just one — `lib/push.js`'s `selectNotifyTargets()` accepts either a single sender or an array. Since a merged turn by definition means every real character just acted, there's usually nobody left to notify anyway; this only matters for a 3-4 hero Manlandia/custom game where some heroes are still non-real (no archetype) and thus never part of the gate at all.
- **Bypassing the wait**: two independent escapes, both skip the gate entirely for that one submission (or, for the toggle, the rest of the session) rather than changing what "real character" means:
  - **Private scenes** (`private: true`, Resonance-only — see "Solo/private scenes" below) bypass automatically. A private scene is inherently one character acting alone; requiring the *other* character to also act first would defeat the entire mechanic.
  - **`soloOverride: true`** in the request body — driven by the "🧍 Solo tonight" toggle (`#solo-mode-toggle`, `setupSoloMode()` in `public/game.js`), a session-long flag mirroring the existing 🎭 Private Scene toggle's UX exactly (click to arm, stays on until clicked again or the page reloads — deliberately not persisted to `localStorage`, so a shared device doesn't silently stay in solo mode for whoever opens the app next). For when a partner genuinely isn't around for the night.
- **Solo worlds are unaffected** — `getRealCharacterKeys(...).length > 1` is the only condition that ever triggers the gate, so a one-hero Manlandia/custom campaign (including the Dark Wars testing ground) behaves exactly as it always did: every action advances immediately.
- **UI**: `renderWaitingBanner()` (`public/game.js`) now checks `worldState.pending_turn` first — if it's non-empty, the banner shows either "⏳ {names} acted — the story continues once you act too!" (when the current viewer is one of the people still needed) or "⏳ Your action is in — waiting on {names}" (once they've submitted and are waiting on someone else), overriding the older purely-informational `last_actor`-based banner underneath, which is still what renders once nobody's action is actually pending.
- **Recovering from the double-roll bug** (see "Session log mechanics" below) and this feature share no code, but both were driven by the same real-world observation: multiple people acting on the same shared world at once is a case this app's original single-submitter model never really accounted for.

## The Living World (ambient "meanwhile..." beats)

A second, richer tier of the turn-stall cron above — instead of just "your turn is waiting," a world stalled long enough gets an actual short atmospheric beat showing it moving on without the players, plus a push notification. Reuses the exact same cron run, Redis keys, and push plumbing as the plain reminder; the only genuinely new piece is a small LLM call made from inside a cron job instead of a player-triggered endpoint.

- **Threshold and sequencing**: `AMBIENT_MS` (72h) in `api/cron-turn-reminder.js` is deliberately longer than `STALL_MS` (48h) — the ambient beat is the next tier of staleness, not a duplicate ping alongside the plain reminder. `checkAmbient()` runs first for each world; if it actually sends a beat, the plain `checkWorld()` reminder is skipped for that world in that run (`if (ambientResult.sent) continue`), so a badly-stalled world never gets two pushes in the same cron pass. Fires once per stall via its own `worldState.last_ambient_sent_at` checkpoint, independent of `last_reminder_sent_at` — same "a fresh action naturally re-arms it" pattern as the plain reminder.
- **Flavor-only by construction, not just by convention**: `lib/livingworld.js`'s `buildAmbientPrompt(worldConfig, gameState)` teaches the model nothing about bracket-tag notation at all — no STATE NOTATION block, no examples. `checkAmbient()` never calls `applyStateTags()` on the result either. This makes state-mutation structurally impossible during an ambient beat, not merely discouraged by prompt wording — the design concern (an unsupervised cron-triggered LLM call silently moving real game state nobody witnessed) is closed off at the code level. Even a stray bracket the model emitted anyway would just get silently stripped for display like any other unrecognized text (`public/pure.js`'s `stripGMTags()`), never parsed.
- **Per-world tone**: `buildAmbientPrompt()` branches on `worldConfig.id`/`type` the same way `buildWorldStatePayload()` does — Resonance gets a Varek/Conclave-flavored ominous beat, Manlandia gets a kid-safe Greying-Curse-flavored beat, and custom worlds get a beat built from whatever `name`/`theme` the player wrote at creation, adult/kid tone split via the same `isAdultWorld()` check used everywhere else.
- **Storage and rendering**: the beat is appended to `sessionLog` as `{ role: "gm", content, ambient: true, timestamp }` — it flows through the exact same `/api/poll` → `appendGMEntry()` path as real narration (no new endpoint, no new tab), just rendered with a "🌙 Meanwhile…" label and a dashed/italic style (`.log-entry.ambient` in `public/style.css`) so it reads as atmosphere, not a real GM turn. Context for the LLM call is the last 10 `sessionLog` entries, mapped the same `role: "gm" → "assistant"` way `api/gm.js` already does.
- **Push**: `lib/push.js`'s `buildAmbientPayload()` — same generic, narration-free stance as every other push payload in this app. Unlike the plain reminder (which excludes the last actor, since they don't need reminding about their own idle turn), the ambient push has no single "sender" to exclude — everyone subscribed gets it, since the world moving on is news to the whole party equally.
- **Scope**: universal — Resonance, Manlandia, and every non-archived custom campaign, same as the plain reminder. A Groq failure mid-generation is caught and simply falls through to the plain reminder for that world in that run, rather than failing the whole cron pass.

## Testing

`npm test` runs `node --experimental-test-module-mocks --test tests/*.test.js` — plain `node:test`/`node:assert`, zero test-framework dependency.

- **Pure modules** (`lib/suggestions.js`, `lib/recap.js`, `lib/worldconfig.js`, `public/pure.js`) are `require()`'d directly — no mocking needed.
- **API handlers** are tested by stubbing `lib/redis.js` (and `lib/gemini.js` where the handler calls the LLM, or `web-push` where it sends a notification) with `t.mock.module()`, then invoking the handler with a fake `req`/`res`. See `tests/helpers.js` for the shared `mockRes()`, `freshRequire()` (clears `require.cache` so a fresh mock takes effect), `statefulRedisMock()` (single-key, persists across calls within a test) and `keyedRedisMock()` (multi-key, for handlers like `api/campaigns.js`/`api/push.js` that touch more than one Redis key per request).
- **`public/game.js`** itself is not unit-tested — it's a DOM-heavy browser script with no module system. The handful of genuinely reusable, logic-only functions it used to contain (`stripGMTags`, `getPlayerDisplayName`, `formatCampaignExport`, etc.) were moved into `public/pure.js`, which is loaded as a plain global-scope script before `game.js` and doubles as a CommonJS module for tests (see the `typeof module !== "undefined"` guard at the bottom of that file, and the `defaultWorld()`/`defaultGameState()`/etc. helpers that let its functions read the browser's globals while still being safely callable with explicit args from Node). Verify DOM-dependent changes by hand or with a real browser — a mock static server + Playwright script was used to smoke-test the app during development; there's no permanent browser test in this repo.
- **`npm run check-drift`** (`scripts/check-tag-drift.js`) is a separate, live-network check — not part of `npm test`, since it hits production and needs `GAME_SECRET`/`ADULT_PIN` env vars (both optional; gated worlds are just skipped without them). It pulls every live campaign's real transcript and prints every bracket tag the GM has actually emitted, for eyeballing against `lib/gm-tags.js`'s expected formats. This is literally how every GM tag-parsing bug in this app was found — run it as a standard part of every testing pass, not just when something seems broken (the model's real-world compliance drifts over time on its own).
